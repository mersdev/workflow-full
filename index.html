Mersdev Workflow Full

Simulating Car Connectivity Consortium (CCC) Digital Key Owner Pairing using SPAKE2+ and Temporal.io Workflows

Project Overview

The mersdev-workflow-full project provides a comprehensive simulation of the owner pairing process critical for Car Connectivity Consortium (CCC) Digital Key functionality. It demonstrates the secure establishment of a relationship between a device (like a smartphone) and a vehicle using the SPAKE2+ (Secure Password Authenticated Key Exchange) protocol. This entire intricate process is orchestrated using Temporal.io workflows, ensuring reliability and visibility.

This simulation involves two core microservices:

  • workflow-device: Represents the end-user's device acting as a digital key.
  • workflow-vehicle: Represents the vehicle's systems participating in the pairing.

The project highlights how complex, stateful interactions in automotive cybersecurity can be managed effectively, focusing on secure key exchange and robust workflow automation for modern vehicle access systems.

System Architecture

The system comprises two main Spring Boot applications that communicate via HTTP, exchanging SPAKE2+ protocol messages encoded in TLV format. Each application leverages its own dedicated Temporal.io cluster for workflow management.

Key Interactions:

  • Internal Full Cycle Test: Each application can run a complete SPAKE2+ pairing simulation internally.
  • Inter-Service Pairing: A realistic flow where services exchange messages step-by-step. workflow-vehicle uses SbodClient and workflow-device uses DkcClient. The x-requestId header is vital for workflow correlation.

Core Technologies

  • Java 17+ with Spring Boot 3+
  • Temporal.io: For resilient workflow orchestration.
  • SPAKE2+ Protocol: For secure password-authenticated key exchange, implemented with BouncyCastle.
  • TLV (Tag-Length-Value): Message encoding (using com.payneteasy.tlv).
  • Feign Clients: For declarative REST API communication.
  • Gradle: Build automation.
  • Docker & Docker Compose: For Temporal.io environment setup.
  • Lombok: To reduce boilerplate code.

Module: workflow-device

Simulates a mobile device (e.g., smartphone as a digital key).

  • Port: localhost:3030
  • Controller: DeviceOEMController handles incoming requests like /startFullOwnerPairingCycle and /receivedFromVehicle.
  • Services:
    • Spake2PlusDeviceService: Device-side SPAKE2+ crypto logic.
    • DkcService & DkcClient: Sends messages to workflow-vehicle.
    • ReceivedFromVehicleService: Processes incoming messages and manages Temporal workflows.
  • Workflows: Spake2PlusDeviceWorkFlow (for inter-service pairing) and Spake2PlusFullWorkFlow (for internal tests).
  • Temporal UI: http://localhost:8081

Module: workflow-vehicle

Simulates a vehicle's head unit or telematics control unit.

  • Port: localhost:3031
  • Controller: VehicleOEMController handles requests like /startOwnerPairing and /sendToVehicle.
  • Services:
    • Spake2PlusVehicleService: Vehicle-side SPAKE2+ crypto logic.
    • SbodService & SbodClient: Sends messages to workflow-device.
    • SendToVehicleService: Initiates pairing workflows and handles incoming messages.
  • Workflows: Spake2PlusVehicleWorkFlow (for inter-service pairing) and Spake2PlusFullWorkFlow (for internal tests).
  • Temporal UI: http://localhost:8080

Owner Pairing Workflows Explained

SPAKE2+ Protocol High-Level Steps:

  1. Password Derivation (Scrypt): Both parties derive keying material (w0, w1) from a shared password and salt.
  2. Request Phase: One party (e.g., Vehicle) initiates with a SPAKE2+ Request Command. The other (Device) responds with its public value X in a SPAKE2+ Request Response.
  3. Verification Phase: The initiator sends its public value Y and evidence M1 in a SPAKE2+ Verify Command. The responder verifies M1, computes its evidence M2, and sends it back in a SPAKE2+ Verify Response. Mutual verification of evidence confirms a shared secret.

Temporal Workflows:

  • Self-Contained Full Cycle Pairing (Spake2PlusFullWorkFlow): Triggered by /startFullOwnerPairingCycle. Runs entirely within one microservice, simulating both device and vehicle roles for internal logic testing.
  • Inter-Service Owner Pairing (Spake2PlusVehicleWorkFlow & Spake2PlusDeviceWorkFlow): A realistic, multi-step communication between workflow-vehicle and workflow-device, orchestrated by Temporal on both sides. Each step (sending request, awaiting response, sending verification, etc.) is managed reliably.

Temporal.io ensures that these complex, potentially long-running pairing sequences are durable, fault-tolerant, and traceable, which is crucial for Car Connectivity Consortium standards.

Message Format: TLV

Communication for SPAKE2+ protocol steps uses Tag-Length-Value (TLV) encoding, a common standard in secure systems and digital key specifications.

  • Implemented using the payneteasy-tlv library.
  • Java classes in com.xdman.workflow_*/model/tlv/ define these structures (e.g., Spake2PlusRequestCommandTlv).
  • Commands often follow APDU-like structures (e.g., 80 30 ... for Request, ... 90 00 for success response).

Setup and Running

Prerequisites:

  • Java JDK 17+
  • Gradle 8.x (via wrapper)
  • Docker & Docker Compose

Temporal Setup:

Each module requires its own Temporal cluster. Use the provided docker-compose-*.yml files:

For workflow-device (in workflow-device/):
docker-compose -f docker-compose-device.yml up -d

For workflow-vehicle (in workflow-vehicle/):
docker-compose -f docker-compose-vehicle.yaml up -d

Build & Run:

In each module's directory (workflow-device and workflow-vehicle):

  1. Build: ./gradlew build
  2. Run: ./gradlew bootRun or java -jar build/libs/*.jar

Testing the Flow

The test.http file in the project root provides HTTP requests for testing various owner pairing scenarios using an IDE plugin (like VS Code REST Client or IntelliJ HTTP Client).

Example Test Cases:

  • Internal Full Cycle Pairing tests for both device and vehicle.
  • Vehicle-initiated Inter-Service Owner Pairing.
  • Simulated message receptions to trigger specific workflow signals.

Monitor application logs and Temporal Web UIs for workflow execution details and status.
